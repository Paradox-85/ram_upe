# Read Repo IDX File List

План для тестовой простой задачи: прочитать текущий репозиторий и создать в `.idx` индекс со списком файлов.

## Context
- Репозиторий находится в `C:/Development/upe/UPE`.
- Текущая структура верхнего уровня: `.git/`, `.plans/`, `docs/`, `prompts/`, `src/`, `azure-pipelines.yml`.
- Папки `.idx` сейчас нет.
- В проекте в основном Markdown/текстовые knowledge-файлы по UPE:
  - `docs/` — executive summary, functional blocks, brainstorming, PDF.
  - `prompts/` — framework prompt `LLM-Native Product Design Framework.md`.
  - `src/chat/`, `src/loop/`, `src/moms/`, `src/vendors/` — исходные заметки, transcripts, vendor материалы.
- Уже есть `.plans/` с предыдущим планом `upe-knowledge-repo-demo`; это planning artifacts, их можно включать или исключать из `.idx` по выбранному правилу.
- В репозитории не найден `.gitignore`, поэтому лучше явно исключать служебные папки `.git/` и, при необходимости, `.idx/` из генерируемого списка.

## Plan:
1. Проверить текущее состояние репозитория read-only командами: `git status --short`, `find`/`ls` для верхнего уровня, убедиться что `.idx/` отсутствует или понять, какие файлы в ней уже есть.
2. Определить простое правило индексации: включать все файлы проекта, кроме `.git/**` и самой `.idx/**`; для теста можно включать `.plans/**`, чтобы индекс отражал весь рабочий репозиторий, либо исключить `.plans/**`, если нужен только product/source контент.
3. Создать папку `.idx/` в корне репозитория.
4. Сформировать основной файл `.idx/files.md` со списком файлов. Рекомендуемый формат:
   - заголовок `# Repository File Index`;
   - дата/время генерации;
   - краткое правило включения/исключения;
   - секции по директориям (`docs/`, `prompts/`, `src/`, `.plans/`, root files);
   - относительные пути в Markdown bullet list.
5. Дополнительно, если нужен максимально простой машинно-читаемый вариант, создать `.idx/files.txt` с одним относительным путём на строку. Для тестовой задачи это опционально; минимально достаточно `.idx/files.md`.
6. Для построения списка использовать стабильную сортировку путей по алфавиту. Не читать содержимое всех больших файлов полностью; задача — индекс путей, не контента.
7. Убедиться, что `.idx/files.md` не включает `.git/**` и не включает сам себя рекурсивно как исходный файл индексации. Если `.idx/files.txt` создаётся, также исключить `.idx/**` из исходного набора.
8. Выполнить sanity check: перечитать `.idx/files.md`, проверить наличие ключевых файлов (`azure-pipelines.yml`, `docs/UPE_Executive_Summary_v1.md`, `prompts/LLM-Native Product Design Framework.md`, `src/loop/loop.md`) и отсутствие `.git/HEAD`.
9. В финальном ответе кратко перечислить, что создано: `.idx/files.md` и, если выбран опциональный шаг, `.idx/files.txt`; указать количество проиндексированных файлов.

## Risks / Open Questions
- Неясно, должен ли индекс включать `.plans/**`. Для теста допустимо включить, но если `.idx` нужен как индекс только продуктовых материалов, `.plans/**` лучше исключить.
- Неясно, нужен ли только human-readable Markdown или ещё plain text/JSON. План предлагает `.idx/files.md` как минимум и `.idx/files.txt` как простой optional artifact.
- PDF и большие transcript/vendor файлы не нужно парсить для этой задачи: индексируется путь к файлу, а не содержимое.
